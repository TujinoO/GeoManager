# 中亚胡杨林生态系统保护数据共享平台软件设计文档

## 1. 文档概述

### 1.1 文档目的

本文档用于描述“中亚胡杨林生态系统保护数据共享平台”的软件总体设计、功能模块、页面组成、数据管理、权限控制、技术栈、配置方式、测试验收和交付要求。

本文档重点说明软件的功能设计和主要技术约束，不展开详细接口设计、数据库表结构设计、部署方案或底层代码实现细节。

### 1.2 软件定位

中亚胡杨林生态系统保护数据共享平台是一套 B/S 架构的网页端软件，面向胡杨林生态系统保护、空间数据管理、地图展示、数据查询、成果展示和数据共享服务。软件应该是中文界面。

用户通过浏览器访问系统。系统采用统一登录入口，登录后进入软件主界面。后台管理不作为登录页中的独立入口，而是作为登录后的功能模块，根据用户权限显示或启用。

### 1.3 建设目标

平台建设目标应从业务目标、能力目标和管理目标三个层级进行组织，不在本章节展开具体页面按钮、缓存策略、脚本调用方式等实现细节。

#### 1.3.1 业务目标

平台应面向中亚胡杨林生态系统保护相关工作，建立统一的数据管理、地图展示、成果共享和业务使用入口，为项目数据查看、查询、展示和后续维护提供支撑。

#### 1.3.2 能力目标

平台应具备对项目相关数据的统一接入、组织、管理、展示、查询和基础分析能力，能够支撑矢量空间数据、栅格空间数据、业务属性数据和成果资料等多类型数据的综合使用。

#### 1.3.3 管理目标

平台应具备用户权限管理、数据资源管理、图层管理、成果管理、系统配置、日志记录和运行维护能力，保证平台可配置、可维护、可扩展。

#### 1.3.4 技术目标

平台应采用适合项目规模和数据特点的 B/S 技术路线，保证前后端协同、数据存储清晰、程序与数据分离、系统配置规范，并为后续数据扩展和功能扩展预留空间。

### 1.4 设计边界

本软件包含以下内容：

1. 网页端地图展示与业务功能。
2. 登录后的后台管理功能。
3. 数据资源、数据目录、图层、成果和元信息管理。
4. 用户、角色和权限管理。
5. 矢量数据前端符号化展示。
6. 栅格数据后台符号化 PNG 出图、缓存和展示。
7. TOML 配置文件加载和关键配置校验。
8. 系统运行日志和关键操作记录。

本阶段不包含以下内容：

1. 原始遥感影像的专业生产处理流程。
2. 大规模遥感算法模型训练。
3. 独立移动端 App。
4. 第三方大型数据中心建设。
5. 复杂办公审批流系统。
6. 专题图制作功能。
7. 用户自定义样式模板功能。

专题图制作和用户自定义样式模板作为后续扩展能力预留，不作为本阶段必须交付内容。

## 2. 总体设计

### 2.1 软件形态

本软件采用 B/S 架构。用户通过浏览器访问系统，完成登录后进入软件主界面。

软件主界面承载业务功能和后台入口。后台入口作为软件主界面中的菜单项或功能按钮，根据用户权限显示或启用。

### 2.2 页面入口关系

系统页面入口关系如下：

1. 用户访问系统地址。
2. 系统展示统一登录页。
3. 用户登录成功后进入软件主界面。
4. 普通用户使用地图浏览、图层管理、数据查询、成果展示等业务功能。
5. 具备后台权限的用户可在软件主界面中进入后台管理功能。
6. 无后台权限的用户不显示后台入口，或入口不可点击并提示无权限。

### 2.3 用户角色

系统用户角色包括：

1. 普通用户：使用地图浏览、图层查看、数据查询、属性查看和成果浏览等基础功能。
2. 科研用户：在普通用户功能基础上，可使用更高权限的数据查询、结果导出、数据元信息查看等功能。
3. 数据管理员：负责数据资源、图层、目录、成果资料和数据元信息维护。
4. 系统管理员：负责用户、角色、权限、系统配置、日志和后台管理功能维护。

如需公开访问能力，可根据项目需要配置游客或公开用户权限。

### 2.4 权限原则

系统应根据用户角色和权限控制页面、菜单、按钮、图层、数据资源和后台功能访问范围。

权限控制原则如下：

1. 未登录用户访问受保护页面时跳转到统一登录页。
2. 登录用户只能访问已授权的功能和数据。
3. 无后台权限用户不得进入后台管理界面。
4. 无权限菜单不显示或不可点击。
5. 无权限图层不显示或不可加载。
6. 无权限数据不返回查询结果。
7. 数据导出、数据维护、系统配置等敏感操作必须进行权限校验。

## 3. 软件主界面设计

### 3.1 登录页

系统应提供统一登录页。登录页不展示独立后台入口。

登录页应包含：

1. 用户账号输入。
2. 用户密码输入。
3. 登录按钮。
4. 登录状态提示。
5. 记住登录状态选项。
6. 忘记密码入口。
7. 用户注册入口，是否开放注册由后台配置决定。

### 3.2 软件主界面

用户登录成功后进入软件主界面。软件主界面是业务功能和后台入口的统一承载页面。

软件主界面应包含：

1. 顶部导航栏。
2. 地图主视图。
3. 图层管理面板。
4. 数据查询面板。
5. 工具栏。
6. 结果展示面板。
7. 图层渲染与可视化面板。
8. 用户信息与退出入口。
9. 后台管理入口，是否显示或启用由用户权限决定。

### 3.3 后台管理界面

具备后台权限的用户从软件主界面进入后台管理界面。

后台管理界面应包含：

1. 后台顶部栏。
2. 左侧菜单栏。
3. 主内容区。
4. 用户信息区。
5. 系统消息与操作反馈区。

后台菜单应根据用户角色和权限动态加载。

------

## 4. 软件功能设计

### 4.1 地图浏览

地图浏览是软件主界面的核心功能。系统应支持用户在浏览器中浏览项目相关空间数据和业务数据。

地图浏览应支持：

1. 地图放大、缩小和平移。
2. 地图旋转和倾斜浏览。
3. 比例尺和经纬度坐标显示。
4. 地图全屏显示。
5. 地图视角复位。
6. 按指定区域或指定坐标定位。
7. 底图切换。
8. 多图层叠加显示。
9. 图层透明度调节。
10. 矢量图层展示。
11. 栅格文件后台生成 PNG 后的图像展示。

地图动态浏览的帧率目标不低于 20fps。

### 4.2 图层管理

图层管理用于对地图中加载的数据图层进行统一管理和控制。

系统应支持：

1. 查看图层列表。
2. 图层分类展示。
3. 图层分组管理。
4. 图层开启和关闭。
5. 图层显隐切换。
6. 图层排序。
7. 图层透明度设置。
8. 图层名称显示。
9. 图层图例显示。
10. 图层元信息查看。
11. 图层快速定位。
12. 图层搜索。

图层分类应支持后台配置，不在程序中固定写死。图层操作响应时间目标不超过 500ms，系统应支持不少于 15 个图层并发加载与管理。

### 4.3 数据加载

数据加载用于将平台已登记的数据资源加载到地图中展示。

系统应支持：

1. 从数据目录选择数据加载。
2. 按图层分类加载数据。
3. 按空间范围或项目范围加载数据。
4. 按时间范围加载数据。
5. 批量加载多个图层。
6. 取消已加载数据。
7. 查看数据加载状态。
8. 加载失败提示。
9. 重新加载数据。

具体数据类型、字段结构、坐标信息、空间范围和展示方式以后续实际数据清单为准。

### 4.4 数据查询

数据查询用于帮助用户从平台数据资源中检索数据对象、属性信息和空间范围。

属性查询应支持：

1. 按关键词查询。
2. 按数据资源查询。
3. 按数据分类查询。
4. 按数据来源查询。
5. 按时间信息查询。
6. 按字段条件查询。
7. 按自定义条件查询。

空间查询应支持：

1. 点选查询。
2. 矩形范围查询。
3. 多边形范围查询。
4. 圆形缓冲区查询。
5. 当前地图范围查询。
6. 指定空间范围查询。

组合查询应支持属性条件、空间范围、时间条件和多字段条件组合。

查询结果应支持：

1. 地图高亮显示。
2. 列表展示。
3. 表格展示。
4. 弹窗展示。
5. 详情页展示。
6. 统计数量展示。
7. 授权范围内的结果导出。

当单次查询数据量不超过 3 万条时，查询响应时间目标不超过 1 秒。

### 4.5 属性详情查看

用户点击地图对象或查询结果对象时，系统应展示对象详情。

详情内容应包括：

1. 对象名称。
2. 对象类型。
3. 所属数据集。
4. 主要属性字段。
5. 空间位置信息。
6. 更新时间。
7. 数据来源。
8. 数据质量说明。
9. 关联图层。
10. 关联附件或图片。
11. 返回地图定位。

### 4.6 矢量符号化渲染

矢量数据符号化由前端基于 Mapbox GL JS 实现。

系统应支持：

1. 点符号显示。
2. 线样式显示。
3. 面填充样式显示。
4. 边界样式显示。
5. 颜色设置。
6. 透明度设置。
7. 分类渲染。
8. 分级渲染。
9. 标签显示。
10. 标签字段选择。

矢量符号化参数应根据实际矢量数据字段、几何类型和前端展示能力确定。

### 4.7 栅格 PNG 展示

栅格文件不在前端直接进行符号化计算。前端仅负责接收后台生成的 PNG 图像，并在地图或结果面板中展示。

前端应支持：

1. 加载后台返回的栅格 PNG 图像。
2. 在地图上叠加显示栅格 PNG。
3. 设置栅格 PNG 图层透明度。
4. 查看栅格图层图例或说明。
5. 查看栅格元信息。
6. 处理 PNG 加载失败提示。

### 4.8 数据目录浏览

数据目录用于组织和展示平台已接入的数据资源。

系统应支持：

1. 按数据分类浏览。
2. 按业务主题浏览。
3. 按数据来源浏览。
4. 按更新时间浏览。
5. 按空间范围或项目范围浏览。
6. 搜索数据资源。
7. 查看数据简介。
8. 查看数据元信息。
9. 加载数据到地图。
10. 查看数据权限状态。

数据分类、业务主题和空间范围应支持后台配置。

### 4.9 成果展示

成果展示用于集中展示项目形成的图件成果、数据成果、说明材料和平台应用成果。

系统应支持：

1. 成果列表展示。
2. 成果分类筛选。
3. 成果详情查看。
4. 成果地图定位。
5. 成果图层加载。
6. 成果附件查看。
7. 成果图片预览。
8. 成果说明文本展示。
9. 成果更新时间展示。
10. 成果来源展示。

### 4.10 统计分析展示

系统应提供基础统计展示能力，用于对查询结果、图层数据和成果数据进行概览展示。

系统可支持：

1. 数据总量统计。
2. 分类数量统计。
3. 区域分布统计。
4. 时间分布统计。
5. 查询结果统计。
6. 图表展示。
7. 统计结果与地图联动。

图表类型可包括柱状图、折线图、饼图、环形图、指标卡片和简单排行表。

### 4.11 个人中心

个人中心用于用户查看和维护个人基础信息。

系统应支持：

1. 查看个人信息。
2. 修改个人基础信息。
3. 修改密码。
4. 查看个人角色。
5. 查看个人权限范围。
6. 查看收藏图层。
7. 查看保存的地图视图。
8. 退出登录。

------

## 5. 后台管理功能设计

### 5.1 后台首页

后台首页用于展示系统管理概览信息，包括用户数量、图层数量、数据资源数量、成果数量、最近登录用户、最近数据更新记录、最近操作日志和系统提示信息。

### 5.2 用户管理

用户管理用于维护平台用户账号和用户状态。

系统应支持：

1. 新增、编辑、删除用户。
2. 启用、禁用用户。
3. 重置用户密码。
4. 查询和筛选用户。
5. 查看用户详情。
6. 分配用户角色。
7. 设置用户数据访问范围。

### 5.3 角色与权限管理

角色与权限管理用于控制不同用户对系统功能、后台菜单、数据资源和操作行为的访问范围。

系统应支持：

1. 新增、编辑、删除角色。
2. 启用、禁用角色。
3. 设置角色名称和描述。
4. 配置角色功能权限。
5. 配置角色数据权限。
6. 查看角色关联用户。
7. 控制后台入口访问权限。
8. 控制图层、数据查询、数据导出、数据维护、系统配置和日志查看权限。

专题图制作和样式模板管理权限作为预留权限项，不作为本阶段必须启用权限。

### 5.4 数据资源管理

数据资源管理用于维护平台接入的数据资源及其基础信息。

系统应支持：

1. 新增、编辑、删除数据资源。
2. 启用、停用数据资源。
3. 批量导入数据。
4. 查看数据资源详情。
5. 配置数据分类、来源、更新时间和说明。
6. 配置数据访问权限。
7. 查看数据状态。
8. 记录数据维护日志。

### 5.5 数据目录管理

数据目录管理用于对平台数据资源进行分类组织。

系统应支持：

1. 新增、编辑、删除目录。
2. 调整目录排序。
3. 配置目录层级和说明。
4. 目录下挂接数据资源。
5. 设置目录访问权限。
6. 启用或停用目录。

### 5.6 图层管理

图层管理用于配置软件主界面中可展示和可加载的地图图层。

系统应支持：

1. 新增、编辑、删除图层。
2. 启用、停用图层。
3. 配置图层名称、分类和显示顺序。
4. 配置图层默认显隐状态和默认透明度。
5. 配置图层图例和说明信息。
6. 配置矢量图层基础符号化样式。
7. 配置栅格图层对应的后台符号化脚本和符号化规则。
8. 配置图层数据来源。
9. 配置图层访问权限。

### 5.7 成果管理

成果管理用于维护平台展示的项目成果、图件、分析结果和相关说明文件。

系统应支持：

1. 新增、编辑、删除成果。
2. 发布、下架成果。
3. 设置成果分类和展示顺序。
4. 上传成果图片。
5. 关联地图图层。
6. 关联附件资料。
7. 编写成果说明。
8. 设置成果访问权限。

### 5.8 系统配置管理

系统配置管理用于维护平台基础运行参数和页面显示配置。

系统应支持：

1. 配置平台名称。
2. 配置首页标题。
3. 配置默认地图中心点。
4. 配置默认地图缩放级别。
5. 配置默认底图。
6. 配置公开访问范围。
7. 配置是否允许用户注册。
8. 配置文件上传限制。
9. 配置查询结果数量限制。
10. 配置系统公告。
11. 查看业务数据总目录和地理数据总目录配置。
12. 查看 TOML 配置文件加载状态。
13. 查看或配置栅格 PNG 缓存大小上限和清理策略。

### 5.9 栅格符号化与缓存管理

栅格符号化与缓存管理用于维护栅格文件后台出图规则、专题栅格脚本配置、PNG 缓存状态和缓存大小上限。

系统应支持：

1. 配置栅格文件对应的专题符号化脚本。
2. 配置栅格符号化规则。
3. 配置 PNG 输出尺寸或缩略图尺寸。
4. 按固定目录规则使用栅格 PNG 输出目录和缓存目录。
5. 查看 PNG 缓存文件数量和总大小。
6. 设置缓存大小上限。
7. 设置缓存清理策略。
8. 手动清理缓存。
9. 查看栅格符号化脚本执行日志。
10. 查看栅格出图失败原因。
11. 重新生成指定栅格 PNG。

当缓存目录总大小超过上限时，系统应按配置的缓存清理策略自动清理缓存文件。

### 5.10 操作日志管理

操作日志用于记录用户在系统中的关键操作，便于追溯、审计和问题排查。

系统应记录：

1. 用户登录和退出日志。
2. 数据新增、编辑、删除日志。
3. 图层配置日志。
4. 权限变更日志。
5. 成果发布日志。
6. 查询和导出日志。
7. 栅格符号化脚本执行日志。
8. 系统配置变更日志。

后台应支持按操作用户、操作类型、操作时间、操作模块、操作结果和关键词查询日志。

### 5.11 字典管理

字典管理用于维护平台运行所需的基础分类和枚举信息。

系统应支持维护：

1. 数据分类。
2. 图层分类。
3. 成果分类。
4. 用户状态。
5. 数据来源。

## 6. 数据与存储设计

### 6.1 数据类型原则

当前项目具体数据类型、数据格式和数据清单尚未完全确定，系统不在设计阶段固定具体数据清单。

平台应具备对以下通用数据形态的管理、展示和扩展接入能力：

1. 矢量空间数据。
2. 栅格空间数据。
3. 表格属性数据。
4. 项目成果数据。
5. 文档、图片、附件等辅助资料。

具体数据名称、数据分类、数据字段、数据格式、空间范围和入库方式应以后续甲方提供的数据资料和双方确认的数据清单为准。

### 6.2 数据元信息

每项数据资源应具备必要元信息，包括：

1. 数据名称。
2. 数据编号。
3. 数据类型。
4. 数据分类。
5. 数据来源。
6. 数据提供单位。
7. 数据时间。
8. 更新时间。
9. 空间范围。
10. 坐标信息。
11. 数据格式。
12. 数据说明。
13. 数据质量说明。
14. 数据访问权限。
15. 维护人员。

### 6.3 数据质量约束

平台应对数据质量进行基础约束：

1. 数据名称应规范。
2. 数据分类应明确。
3. 空间数据应具备正确的空间范围。
4. 重要属性字段应完整。
5. 数据来源应可追溯。
6. 数据更新时间应明确。
7. 无效数据应可停用或标记。

### 6.4 存储分离原则

平台应遵循“程序本身、业务数据、地理数据分离”的原则。

1. 程序目录只保存系统代码、Django 项目、前端资源、Python 依赖、栅格符号化脚本和运行程序。
2. 业务数据使用独立的业务数据总目录保存。
3. 地理数据使用独立的地理数据总目录保存。
4. 业务数据总目录和地理数据总目录通过 TOML 配置文件设定。
5. 两个总目录下的数据组织结构固定，不在配置文件中逐项配置每个子目录。

### 6.5 存储方式

平台数据按以下方式存储：

1. 一般业务数据库使用 SQLite，存放在业务数据总目录下的固定数据库目录中。
2. 业务附件、导出文件、运行日志等业务运行数据存放在业务数据总目录下的固定子目录中。
3. 矢量地理数据使用 GeoPackage，文件格式为 `.gpkg`，存放在地理数据总目录下的固定矢量数据目录中。
4. 栅格数据以文件形式直接存储，存放在地理数据总目录下的固定栅格数据目录中。
5. 栅格符号化后生成的 PNG 文件作为输出文件或缓存文件，存放在地理数据总目录下的固定 PNG 目录中。

### 6.6 固定目录组织

业务数据总目录和地理数据总目录由 TOML 配置文件指定。两个总目录下的子目录采用固定约定，便于程序统一读取、维护、备份和迁移。

系统不得将业务附件、数据库文件、GeoPackage、原始栅格文件、栅格 PNG 输出或缓存文件与程序代码混合存放。

## 7. 栅格符号化与缓存设计

### 7.1 处理原则

栅格文件不在前端进行符号化处理。系统应在后台通过专用 Python 脚本读取栅格文件，并根据符号化规则生成 PNG 图像后提供给前端展示。

### 7.2 处理流程

栅格出图流程如下：

1. 前端请求栅格图层展示。
2. 后台根据栅格文件、符号化规则和输出尺寸生成缓存标识。
3. 后台检查是否存在可用缓存 PNG。
4. 若缓存命中，直接返回 PNG 访问地址。
5. 若缓存不存在或失效，后台调用对应专题栅格符号化脚本。
6. 符号化脚本读取原始栅格文件并生成 PNG。
7. 后台记录 PNG 文件路径、生成时间、文件大小、关联数据资源和执行状态。
8. 后台向前端返回 PNG 访问地址。
9. 前端加载 PNG 并在地图或结果面板中展示。

### 7.3 专题脚本机制

栅格文件符号化通过专用 Python 脚本实现。每个专题栅格文件可配置单独的符号化脚本。

系统应预留统一调用接口，用于屏蔽不同专题脚本之间的差异。统一接口应支持标准输入和标准输出，便于后台统一调度、日志记录、错误处理和缓存管理。

统一输入建议包括：

1. 栅格文件路径。
2. 输出 PNG 文件路径。
3. 符号化规则配置。
4. 缩略图尺寸或输出尺寸。
5. 数据范围或裁剪范围。
6. 透明背景配置。
7. 其他专题脚本扩展参数。

统一输出建议包括：

1. 执行状态。
2. 输出 PNG 文件路径。
3. 输出图像宽度和高度。
4. 栅格值范围或统计信息。
5. 错误信息。
6. 执行耗时。

### 7.4 缓存机制

系统应为后台生成的栅格 PNG 图像提供缓存功能，避免同一栅格文件和同一符号化规则被重复生成。

缓存功能应支持：

1. 根据栅格文件、符号化规则、输出尺寸等信息生成缓存标识。
2. 优先读取已有缓存 PNG。
3. 缓存不存在或失效时重新生成 PNG。
4. 记录缓存文件路径、生成时间、文件大小和关联数据资源。
5. 支持手动清理缓存。
6. 支持自动清理缓存。
7. 支持设置缓存大小上限。
8. 当缓存总量超过上限时，按配置策略清理旧缓存。

缓存清理策略可采用最近最少使用、按生成时间清理、按访问时间清理或按文件大小清理等方式，具体策略由 TOML 配置文件或后台配置项确定。

## 8. TOML 配置文件设计

### 8.1 配置原则

系统应提供完整的后台配置文件，配置文件采用 TOML 格式。

TOML 配置文件用于集中维护系统基础配置、数据总目录配置、缓存配置和脚本接口配置。

配置文件应体现程序、业务数据和地理数据分离的原则。程序目录不得作为业务数据总目录或地理数据总目录使用。

### 8.2 配置内容

TOML 配置文件应至少包含：

1. 系统名称。
2. 运行模式。
3. 业务数据总目录。
4. 地理数据总目录。
5. 栅格 PNG 缓存大小上限。
6. 栅格缓存清理策略。
7. 文件上传大小限制。
8. 默认地图中心点。
9. 默认地图缩放级别。
10. 栅格符号化统一接口相关配置。
11. 其他后续扩展参数。

业务数据总目录和地理数据总目录下的子目录组织由系统固定，不在 TOML 配置文件中逐项配置。

### 8.3 配置校验

系统启动时应读取 TOML 配置文件，并对关键配置项进行校验。

校验内容包括：

1. 配置文件是否存在。
2. 配置格式是否正确。
3. 业务数据总目录是否有效。
4. 地理数据总目录是否有效。
5. 目录是否具备必要读写权限。
6. 缓存大小上限是否合理。
7. 固定子目录是否存在。
8. 栅格符号化统一接口配置是否有效。

固定子目录不存在时，系统可根据配置策略自动创建，或记录错误并提示管理员处理。

## 9. 技术栈说明

### 9.1 前端技术

前端采用以下技术：

1. React：用于构建网页端应用界面、组件状态管理和页面交互逻辑。
2. Vite：作为前端工程构建工具，用于项目开发、调试、构建和打包。
3. Mapbox GL JS：用于网页端地图展示、矢量图层渲染、空间交互和 PNG 图像展示。
4. Ant Design：用于实现除地图渲染外的主要 UI 组件，包括登录页、导航栏、菜单、按钮、表格、表单、弹窗、抽屉、标签页、后台管理页面和操作反馈等。
5. HTML、CSS、JavaScript 或 TypeScript：用于网页界面基础开发和工程组织。

前端负责地图展示、矢量图层渲染、后台 PNG 图像展示以及页面 UI 交互，不负责栅格文件符号化计算。

前端界面设计应遵循以下原则：

1. 地图相关能力以 Mapbox GL JS 为核心实现。
2. 普通业务界面和后台管理界面优先使用 Ant Design 组件实现。
3. 页面结构、表格、表单、弹窗、提示信息和后台管理布局应保持风格统一。
4. React 组件应围绕地图展示、图层管理、数据查询、后台管理等功能模块进行组织。
5. Vite 工程应支持前端开发调试、生产构建和静态资源打包。

### 9.2 后端技术

后端采用以下技术：

1. Python：作为后端主要开发语言。
2. Django：作为后端 Web 框架，用于实现业务管理、用户认证、权限控制、后台管理、数据接口和系统基础功能。
3. GeoPandas：用于矢量地理数据读取、处理、转换和空间数据分析支持。
4. GDAL：用于地理空间数据格式读取、转换和基础处理支持。
5. Rasterio：用于栅格数据读取、元信息提取和栅格文件处理支持。

后端负责业务管理、权限校验、数据资源管理、GeoPackage 读取、栅格文件读取、专题栅格脚本调度、PNG 生成、缓存管理和 TOML 配置加载。

### 9.3 Django 能力使用原则

在满足项目功能的前提下，应优先使用 Django 自带或 Django 生态中成熟稳定的能力实现基础功能，避免重复开发。

可优先使用 Django 能力的功能包括：

1. 用户账号管理。
2. 用户登录与退出。
3. 用户认证。
4. 用户组管理。
5. 角色与权限控制。
6. 后台管理页面基础能力。
7. 表单校验。
8. 数据模型管理。
9. 会话管理。
10. 基础安全防护能力。
11. 日志和管理操作记录。

项目特有功能，如 GeoPackage 矢量数据读取、栅格后台符号化出图、PNG 缓存管理和专题脚本调度，可根据业务需要定制开发。

## 14. 总结

中亚胡杨林生态系统保护数据共享平台是一套面向生态保护、空间数据管理和科研成果共享的网页端软件。

系统采用统一登录入口，用户登录后进入软件主界面。后台作为登录后的一个功能模块，根据用户角色和权限显示或启用。

软件主界面侧重地图浏览、图层管理、数据查询、矢量前端符号化渲染、栅格后台 PNG 出图展示和成果浏览。后台侧重用户权限、数据资源、图层配置、栅格符号化脚本配置、PNG 缓存管理、TOML 配置文件管理、成果发布和系统维护。

系统应将程序本身、业务数据和地理数据分离存放，通过 TOML 配置文件设定业务数据总目录和地理数据总目录，并在两个总目录下采用固定子目录组织。具体数据类型和数据清单应以后续实际资料为准。
